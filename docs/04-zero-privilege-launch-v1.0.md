# ProofMoney Zero-Privilege Launch v1.0

**Version:** v1.0  
**Project:** ProofMoney  
**Initial Author:** Vingen Motoki  
**Positioning:** An open protocol for verifiable money integrity  
**Core Statement:** If money cannot be verified, it is only a promise.

---

## 1. Purpose

The Zero-Privilege Launch framework defines how ProofMoney should begin without monetary privilege.

A money integrity protocol cannot begin with hidden advantages.

ProofMoney therefore follows the principle:

> **No participant should receive monetary privilege before public rules exist.**

Zero-Privilege Launch is not a marketing label.

It is a launch constraint.

---

## 2. The Five Zeroes

Zero-Privilege Launch is defined by five zeroes.

### 2.1 Zero Initial Allocation

The starting state must not allocate PRM to any private party.

There should be no initial allocation to founders, core contributors, advisors, investors, insiders, market makers, exchanges, influencers, community groups, ecosystem partners, or private addresses controlled by the project team.

### 2.2 Zero Private Sale

ProofMoney does not sell PRM before public network rules exist.

There should be no seed sale, private sale, pre-sale, insider sale, discounted allocation, whitelist purchase, early access round, advisor round, influencer allocation, or future PRM purchase agreement.

### 2.3 Zero Hidden Address

ProofMoney must not include hidden project-controlled balances.

Any protocol-level public fund must be visible, documented, and governed by public rules.

### 2.4 Zero Yield Promise

ProofMoney does not promise profit, return, income, appreciation, yield, liquidity, or exchange access.

PRM may have no market value.

### 2.5 Zero Rule Privilege

No person, team, foundation, company, or private group should have unilateral authority to change monetary rules.

---

## 3. Starting State Proof

ProofMoney should provide a Starting State Proof.

The Starting State Proof should allow anyone to verify:

- the initial supply;
- whether any spendable PRM exists at launch;
- whether any address received privileged allocation;
- the initial rule version;
- the starting parameters;
- the source code used to generate the starting state.

Expected output:

```text
Initial Supply: 0 PRM
Privileged Allocation: None
Private Sale Allocation: None
Hidden Address Allocation: None
Public Proof Fund Preload: None
Rule Version: v1
Status: Valid
```

---

## 4. Test Environment Rules

Any ProofMoney test environment must be clearly labeled as experimental.

Test units:

- have no monetary value;
- are not PRM;
- are not redeemable for PRM;
- do not create future allocation rights;
- should not be bought or sold;
- do not represent investment rights;
- do not guarantee future participation benefits.

Testing is for improving protocol integrity, not for creating financial expectations.

---

## 5. Team Participation

The ProofMoney initiating author may build, operate, test, document, and support the protocol.

The initiating author may not claim special monetary rights.

Principle:

> **The initiating author can contribute to the protocol, but must not sit above the protocol.**

---

## 6. Launch Integrity Report

Before any live environment is presented as public, ProofMoney should publish a Launch Integrity Report.

The report should include:

- starting state hash;
- initial supply;
- privileged allocation status;
- private sale status;
- hidden address status;
- source code version;
- rule version;
- release curve version;
- Public Proof Fund rule;
- verification command outputs;
- known risks;
- unresolved issues;
- security review status.

---

## 7. Failure Conditions

A launch should not be considered Zero-Privilege if any of the following occur:

- PRM is sold privately before public release;
- founders receive undisclosed initial allocation;
- insiders receive special conversion rights;
- test units are promised future value;
- starting state contains hidden spendable balances;
- public code differs from launched code without disclosure;
- rules are changed privately;
- supply verification tools are unavailable;
- risk disclosures are missing;
- official communication promises returns.

---

## 8. Summary

Zero-Privilege Launch means:

- no initial private allocation;
- no private PRM sale;
- no hidden address;
- no yield promise;
- no unilateral rule privilege;
- no test-unit value promise;
- no launch state that users cannot verify.

ProofMoney does not ask users to trust that the launch is clean.

It must let them verify it.

## Project Safety Notice

ProofMoney does not offer, sell, or solicit the purchase of PRM. PRM does not guarantee price, liquidity, yield, profit, or exchange access. ProofMoney is experimental and may fail. Test units, if any, have no monetary value.
